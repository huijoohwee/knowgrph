## Floating Panel Traversal JSON Cheatsheet

For tests and demos that need to script the Graph Traversal floating panel, this minimal JSON block collects the shared traversal delay, collapse keys, and floating‑panel event into a single place. A nested `traversalQuery` triplet keeps the generic traversal parameters (`traversalStartNodeId`, `traversalMaxDepth`, `traversalLabelFilter`) grouped for test harnesses:

```json
{
  "floatingPanelTraversal": {
    "eventType": "kg:floatingPanelOpen:graphTraversal",
    "orchestratorTraversalDelayMs": 900,
    "traversalQuery": {
      "traversalStartNodeId": "",
      "traversalMaxDepth": 2,
      "traversalLabelFilter": ""
    },
    "collapse": {
      "orchestratorGraphRagCollapsed": true,
      "orchestratorPresetsCollapsed": true,
      "orchestratorEditorCollapsed": true,
      "orchestratorContextCollapsed": true,
      "orchestratorWorkflowIndexingCollapsed": true,
      "orchestratorWorkflowTracingCollapsed": true
    }
  }
}
```

Defaults match the catalog rows above; tests can override any field as needed before dispatching the `eventType` as a `CustomEvent` on `window`. The `traversalMaxDepth` field mirrors `TRAVERSAL_MAX_DEPTH_DEFAULT = 2` and is clamped by `clampTraversalMaxDepth` to the shared `[1,8]` range used in the Render Traversal Query and Orchestrator presets. When scripted tests also tune AI‑KG layout for traversal runs, they should keep `layout.forces.collisionByType` values within the `[4,40]` range enforced by `clampCollisionRadius` (default `COLLISION_RADIUS_DEFAULT = 10`) so expectations match the **AI KG Layers & Traversal** panel and 3D layout physics.

FloatingPanel Graph Traversal also mirrors the Orchestrator traversal key/value layout contract for the interactive “Traversal Query” controls. The “Start node id”, “Max depth”, and “Edge labels filter” inputs are rendered as compact `keyIconValue` rows with right‑aligned `RightAlignedTooltipInput` controls in the 3‑column grid:

- Key (49%) with a dedicated tooltip describing default behavior and impact (for example, “Changes where traversal starts in graph.”).
- Thin icon column (1%) reserved for optional per‑row icons.
- Value (50%) containing a right‑aligned text/number input that shares the same 12px typography and border treatments as GraphRAG workflow dataset and chunking rows.

This keeps Traversal Query inputs visually aligned with `chunking.method`, dataset paths, and embedding configuration rows while preserving the shared zero‑to‑one traversal UX across FloatingPanel and the Orchestrator bottom‑panel UI editor.

## Zero-to-one traversal setup (AI‑KG example)

- Confirm **AgenticRAG context**: in the Orchestrator GraphRAG workflow panel, verify that the `agenticContext.graphContextUrl` row matches the canonical AgenticRAG JSON‑LD context so GraphRAG workflows, generic traversals, QA, and AI‑KG rendering all share the same vocabulary.
- Anchor the traversal query: use Orchestrator presets or the Floating Panel Graph Traversal UI to populate `traversalStartNodeId`, `traversalMaxDepth`, and (optionally) `traversalLabelFilter`, then use the **Run Traversal** button or AI‑KG traversal presets so Orchestrator, Render, and tests all walk from the same starting concept with the same hop limit.
- Tune AI‑KG layers: in the Render tab’s **AI KG Layers & Traversal** block, adjust **Collision Radius** (writes `layout.forces.collisionByType` for catalog node types), **Node Size Formula** (toggles `schema.three.nodeSizingFormula` between `schema` and `importance`), and **Edge Width Formula** (toggles `schema.three.edgeWidthFormula` between `schema` and `weight`) so AI‑KG bands, node prominence, and relation thickness visually match the D3 prototype while replaying the same traversal paths used by GraphRAG and the test suite.
