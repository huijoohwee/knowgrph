# Knowgrph Design Document

## Consistent UI & Icon Group
- Shared header icon group: `canvas/src/features/panels/ui/HeaderActions.tsx` (Search, Pin, Apply, Reset, Minimize/Restore, Close)
- Adopted by Panel (`src/features/panels/Panel.tsx`) and Bottom Panel (`src/components/BottomPanel.tsx`) for consistent ordering and behavior
- Search input aligns to the left of the icon group in headers; toggled via the Search icon
- Floating panel action titles (Minimize/Restore/Close/Pin/Unpin) are centralized in `UI_COPY` (`canvas/src/lib/config.ts`) and reused by the toolbar floating panel and MainPanel header actions.
- MainPanel uses the same elevated container treatment as the floating panel: `rounded-xl border border-gray-200 bg-white shadow-lg shadow-gray-200/60` (`canvas/src/features/panels/ui/MainPanelContainer.tsx`).

### Icon System
- Icon size is driven by the `uiIconScale` setting and the shared `getIconSizeClass` helper (`canvas/src/lib/ui/icons.ts`), so header, toolbar, search, history, Launch Spotlight status, Help/Workflow headers, Graph Fields pills, node/edge tooltip field-type icons, and bottom panel affordances stay visually aligned when switching between compact and default modes. The Help panel icons section (`canvas/src/features/panels/views/HelpSections.tsx`) includes a small legend for Graph Fields type glyphs and notes that node/edge hover tooltips reuse the same icons based on schema property types, instead of exposing separate dev-only uiIconScale toggles.
- Stroke weight is driven by the `uiIconStrokeWidth` setting in the shared UI slice and passed through to `strokeWidth` on Lucide-style SVGs in Toolbar, Sidebar trigger, bottom panel toolbars, header actions, search/history panels, and the Graph Fields icon legend (info glyph plus scope/origin/visibility/type icons) so those badges match toolbar icon weight (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/components/Toolbar.tsx`, `canvas/src/components/SidebarTrigger.tsx`, `canvas/src/components/HistoryPanel.tsx`, `canvas/src/components/SearchPanel.tsx`, `canvas/src/features/panels/ui/HeaderActions.tsx`, `canvas/src/features/toolbar/ToolbarToolMenu.tsx`, `canvas/src/features/toolbar/ToolbarMenuLauncher.tsx`, `canvas/src/features/panels/views/HistoryView.tsx`, `canvas/src/features/panels/ui/MainPanelHelpHeader.tsx`, `canvas/src/features/panels/ui/MainPanelSettingsHeader.tsx`, `canvas/src/features/panels/views/HelpSections.tsx`).
- Icon formatting and color are governed by `uiIconFormat`, `uiIconColorClass`, `uiIconHoverBgClass`, and `uiIconButtonPaddingClass`, which feed into the shared `IconButton` component and keep 3D Mode, Launch Spotlight, sidebar, and panel header actions visually consistent. `uiIconFormat` defaults to `"1"` (a “default styling” sentinel) and can be set to `"minimal"` for stripped hover/ring treatment (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/components/IconButton.tsx`, `canvas/src/components/Toolbar.tsx`, `canvas/src/components/SidebarTrigger.tsx`).
- Icon animation is governed by `uiIconAnimationEnabled`, which controls whether the shared `LaunchButton__icon` CSS animation class is applied to the 3D Mode and Launch toolbar icons. When disabled, icons retain the same size, stroke, and color presets but drop the periodic launch motion (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/components/Toolbar.tsx`, `canvas/src/index.css`).
- Icon coverage includes Graph Data Table filter/sort/group panels, the bottom-panel curator toolbar, Floating Panel workspace actions, Parser tab header actions and info glyphs, and History tab actions. These surfaces route all Lucide and custom SVG icons through the same `getIconSizeClass`/`uiIconStrokeWidth` wiring and `IconButton` affordances, so they visually match the 3D Mode toolbar icons instead of hand-rolled size or stroke settings.
- Main panel presets: the main panel Settings header exposes an “Icons” control strip with **Default**/**Compact** density toggles, a **Motion** toggle (wrapping `uiIconAnimationEnabled`), and a **Compact preset** button that sets compact density and motion together. This gives teams a one-click preset that matches the 3D Mode toolbar styling while still mapping cleanly onto the underlying `uiIconScale` and `uiIconAnimationEnabled` settings.
- Icon legend pills use the `uiIconPillClass` setting persisted in the shared UI slice and exposed under the “UI Density: Icons” area in Panel Settings, so scope, origin, visibility, and field-type chips share a single Tailwind shell across the Help panel legend, Graph Fields list (“Base” badge plus field-type, origin, and visibility chips), AgenticRAG Graph Fields header badges, AgenticRAG node inspector chips, Launch Spotlight status badges, main-panel Graph Fields header chips, and the bottom-panel Orchestrator traversal badge, with additional schema or Graph Data Table badges opting into the same shell over time (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/features/panels/views/HelpSections.tsx`, `canvas/src/features/panels/views/graph-fields/GraphFieldsListPanel.tsx`, `canvas/src/features/panels/ui/MainPanelGraphFieldsHeader.tsx`, `canvas/src/features/panels/ui/StatusBadge.tsx`, `canvas/src/features/panels/views/OrchestratorSettingsSection.tsx`, `canvas/src/features/panels/views/AgenticRagNodeInspectorSection.tsx`).
- Icon legend pills and small AgenticRAG/context badges share a configurable text-size helper: `getPillClass` in `canvas/src/lib/ui/icons.ts` computes Tailwind class strings for `legend` and `badge` variants using `uiIconPillLegendTextSizeClass` and `uiIconPillBadgeTextSizeClass` from the shared UI slice. Panel Settings previews these sizes with Graph Fields-style icon+label pills: the legend text size control renders small “Base”, “Origin”, and “Visibility” pills that combine simple scope/origin/visibility glyphs with text using the legend variant, while the badge text size control renders a compact “Base” icon+label pill using the badge variant so teams can see how Graph Fields header badges and inline “Base” chips will look. Interactive JSON-LD/AgenticRAG chips use the sibling `getChipClass` helper (also in `canvas/src/lib/ui/icons.ts`) to standardize small bordered toggles in Parser sections, Orchestrator text editors, AgenticRAG node inspector hop lists, and Workflow JSON-LD context editors while still accepting per-surface base/text color classes from the shared UI slice. This keeps Graph Fields “Base” badges, Help panel pipeline stage badges, AgenticRAG node inspector hop/traverse chips, Render tab dataset inspector badges, Orchestrator traversal legend settings and AgenticRAG context badges, AgenticRAG context “Ignore filters” badges, Parser Data AgenticRAG edge-context badges, and Workflow tab AgenticRAG workflow context badges visually aligned and tunable from the Panel Settings “UI Density: Icons” section (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/graph-fields/GraphFieldsListPanel.tsx`, `canvas/src/features/panels/views/HelpSections.tsx`, `canvas/src/features/panels/views/AgenticRagNodeInspectorSection.tsx`, `canvas/src/features/panels/views/DatasetInspectorSection.tsx`, `canvas/src/features/panels/views/OrchestratorTextEditorSection.tsx`, `canvas/src/features/panels/views/AgenticRagContextSection.tsx`, `canvas/src/features/panels/views/ParserSections.tsx`, `canvas/src/features/panels/views/WorkflowSteps.tsx`).
- Small schema/property badge tokens use a dedicated badge-chip helper: `getBadgeChipClass` in `canvas/src/lib/ui/icons.ts` composes a compact rounded shell with per-variant text, border, and background color classes while honoring the `uiIconBadgeChipClass` and `uiIconBadgeChipTextSizeClass` settings from the shared UI slice. Graph hover tooltips, the Node editor, and the Advanced schema editor reuse this helper to render schema/property badges and compact property-type badges so teams can tighten or loosen badge density globally via Panel Settings without hand-editing Tailwind strings on each surface (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/components/GraphHoverTooltip.tsx`, `canvas/src/components/NodeEditor.tsx`, `canvas/src/features/schema-editor/AdvancedSection.tsx`). JSON-LD context edge tokens and traversal path node buttons continue to use bordered interactive chips via `getChipClass` instead of reusing the legend pill shell, keeping static legend badges, interactive toggles, and inline schema badges as three related but distinct primitives.
- 3D color settings share a two-input, preview-first control pattern in Panel Settings and 3D tuning panels: `three.camera.backgroundColor`, `three.camera.fogColor`, and `three.graph.starfieldColor` are edited via a color swatch input on the left and a free-form text input on the right. The swatch always receives a normalized hex value (falling back to `#020617`, `#1e1b4b`, or `#facc15` when the stored string is empty or invalid), while the text input preserves the exact string written to the schema/store. Both inputs share a compact, matching height and the swatch uses a fixed `w-8 h-6` footprint, so the preview column and value column sit on a single aligned row. The text input reuses the shared `uiPanelKeyValueInputClass` shell so color strings inherit the same padding, border, and text size as numeric key/value inputs, instead of introducing a separate typographic treatment. This keeps the MainPanel Settings tab, FloatingPanel Three tuning controls, and bottom/side panel 3D controls visually aligned while still allowing teams to type or paste arbitrary color strings when needed (`canvas/src/features/settings/ui.tsx`, `canvas/src/features/panels/views/ThreeViewTuningSection.tsx`, `canvas/src/features/settings/registry-three.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/features/three/Scene.tsx`).
- Panel Settings surfaces preview helpers for icon-related Tailwind settings using the same preview-first two-input pattern and sizing as the 3D color controls. `uiIconColorClass` and `uiIconHoverBgClass` render as paired controls: a fixed-width preview input on the left that applies the configured text color or hover background classes to small “Aa” and “Hover” samples, and a Tailwind class input on the right that edits the raw class string; `uiIconButtonPaddingClass` renders with a Lucide gear icon hit target in a fixed `w-8 h-6` preview cell on the left and a padding-class input on the right; `uiIconPillClass` renders a pill-shaped Scope preview in the same fixed preview footprint to the left of the pill-shell class input; `uiIconPillLegendTextSizeClass` and `uiIconPillBadgeTextSizeClass` render Graph Fields-style icon+label pills wired through `getPillClass` in a left-hand preview block with the text-size Tailwind input on the right; and `uiIconBadgeChipClass` / `uiIconBadgeChipTextSizeClass` render compact badge-chip previews that mirror the schema/property badge shell using a Lucide tag glyph and “Badge” label inside a fixed `w-8 h-6` preview cell on the left, with the corresponding shell/text-size inputs on the right. All of these right-hand Tailwind class inputs reuse the shared `uiPanelKeyValueInputClass` shell so icon density controls share typography, padding, and alignment with numeric panel key/value inputs and 3D color text fields. This preview-first `preview | value` layout is reused for schema editor color controls and label color settings, so color swatches, icon density previews, and text fields across MainPanel Settings, Three tuning panels, and schema style editors all share a single aligned preview column and a single aligned value column (`canvas/src/features/settings/ui.tsx`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/features/panels/views/ThreeViewTuningSection.tsx`, `canvas/src/features/schema-editor/AdvancedSection.tsx`, `canvas/src/components/IconButton.tsx`, `canvas/src/lib/ui/icons.ts`).

- Visual regression checklist for icon density and legend alignment: when tweaking `uiIconColorClass` / `uiIconHoverBgClass`, verify that the Settings preview strip for icon color and hover background uses the same fixed-width preview input footprint and two-input alignment as the `three.camera.backgroundColor` control and that the applied text/hover classes still match IconButton styling in the main toolbar and Graph Fields header; when adjusting `uiIconPillClass` or legend/badge text sizes, confirm that the Settings Scope/Origin/Visibility pills match the Graph Fields panel legend and Help icon legend spacing, font weight, and icon shapes; and when changing badge-chip settings, confirm that Settings badge previews match schema/property badges in Graph hover tooltips and the Node editor.

### Panel key/value text sizing and monospace alignment
- Small panel key/value rows (for example, Traversal delay sliders, Render tuning controls, GraphRAG context inputs, and Schema tuning sliders) share a global text-size setting: `uiPanelKeyValueTextSizeClass` in the shared UI slice (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`). This setting is persisted via `LS_KEYS.panelKeyValueTextSizeClass` (`canvas/src/lib/config.ts`) and exposed in Panel Settings as a free-form Tailwind class string under a “UI Density: Panels” responsibility in the Settings view (`canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`).
- The `KeyTypeValueRow` and `SimpleKeyValueRow` primitives (`canvas/src/features/panels/ui/KeyTypeValueRow.tsx`) read `uiPanelKeyValueTextSizeClass` via `useGraphStore` and apply it to the root grid container for all `layout` variants (`keyTypeValue`, `keyValue`, `keyIconValue`, and `keyIconSliderInput`), so panel rows across the Orchestrator, Render, Schema, Help, and Graph Fields panels stay visually aligned when teams tweak density. Inner spans and controls rely on this inherited font size instead of hardcoding `text-sm` or `text-xs` locally.
- The default value is `text-sm`, chosen to match the Settings input control size. Teams can opt into denser rows (for example, `text-xs` or `text-[11px]`) or more spacious typography (for example, `text-base`) by editing the `uiPanelKeyValueTextSizeClass` setting, without editing individual panel components. Right-aligned numeric text inputs and textarea editors continue to control their own font sizes locally when they must diverge from the panel row default.
- Right-aligned numeric inputs in panel key/value rows share a global input shell setting: `uiPanelKeyValueInputClass` in the shared UI slice (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`). This setting is persisted via `LS_KEYS.panelKeyValueInputClass` (`canvas/src/lib/config.ts`) and exposed in Panel Settings as a free-form Tailwind class string under the same “UI Density: Panels” responsibility (`canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`).
- The `RightAlignedTooltipInput` helper and direct numeric `<input>` controls in dense panel grids (for example, Orchestrator traversal delay, GraphRAG workflow chunk size and max hops, and AI-KG layer opacity/charge/collision sliders) read `uiPanelKeyValueInputClass` via `useGraphStore` and apply it as the base Tailwind shell (`w-full h-6 px-2 text-sm border border-gray-300 rounded text-right` by default). Panel Settings reuses this same shell for all `*Class` and other Tailwind-style text settings (for example, `uiIconColorClass`, `uiIconHoverBgClass`, `uiIconButtonPaddingClass`, `uiIconPillClass`, `uiIconPillLegendTextSizeClass`, `uiIconPillBadgeTextSizeClass`, `uiIconBadgeChipClass`, and `uiIconBadgeChipTextSizeClass`), so class strings and numeric values share a consistent input footprint and typography. Teams can tighten or loosen numeric and class-string input padding, borders, and typography globally by editing this setting instead of hand-editing class strings in each panel component (`canvas/src/features/panels/ui/KeyTypeValueRow.tsx`, `canvas/src/features/panels/views/GraphRagWorkflowSection.tsx`, `canvas/src/features/panels/views/GraphRagWorkflowIndexingSection.tsx`, `canvas/src/features/panels/views/AiKgLayersSection.tsx`, `canvas/src/features/settings/ui.tsx`).
- Panel row padding and spacing share a global density setting for `KeyTypeValueRow`/`SimpleKeyValueRow`: `uiPanelRowDensityDefaultClass` in the shared UI slice controls the Tailwind padding class applied whenever `density="default"` is passed (defaulting to `py-1`). This setting is persisted via `LS_KEYS.panelRowDensityDefaultClass` and exposed under the same “UI Density: Panels” area in Panel Settings so teams can tighten or loosen default row spacing across MainPanel settings, workflow sliders, Graph Fields settings, and related key/value grids without editing each panel component. Compact rows continue to use a separate `uiPanelRowDensityCompactClass` helper (defaulting to `py-0.5`) for surfaces that explicitly opt into `density="compact"` (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`, `canvas/src/features/panels/ui/KeyTypeValueRow.tsx`).
- Floating panels that reuse `KeyTypeValueRow`/`SimpleKeyValueRow` (for example, Graph Traversal presets, Orchestrator traversal sequence inspectors, and AgenticRAG node inspector sections) automatically pick up the same `uiPanelKeyValueTextSizeClass`, `uiPanelKeyValueInputClass`, and `uiPanelRowDensityDefaultClass`/`uiPanelRowDensityCompactClass` settings, so MainPanel, bottom panel, and FloatingPanel key/value grids stay visually aligned without duplicating density or input-shell wiring (`canvas/src/features/panels/views/OrchestratorTraversalPanels.tsx`, `canvas/src/features/panels/views/OrchestratorTraversalSequenceSection.tsx`, `canvas/src/features/panels/views/AgenticRagNodeInspectorSection.tsx`).
- Monospace text in panels and bottom-panel editors shares a single configurable Tailwind class: `uiPanelMonospaceTextClass` in the shared UI slice, persisted via `LS_KEYS.panelMonospaceTextClass` and surfaced under “UI Density: Panels” in Panel Settings (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`). AgenticRAG inspectors, Orchestrator traversal badges, GraphRAG workflow summaries, Graph Data Table ID and numeric cells, JSON editors, and bottom-panel code/parser/schema textareas all read this class via `useGraphStore` and apply it to IDs, IRIs, graphRAGPath chains, JSON snippets, and schema/config text instead of hardcoding `font-mono`. This keeps schema-bound identifiers and JSON-LD snippets typographically aligned across MainPanel, FloatingPanel, and bottom panel views while still allowing teams to switch between compact (`font-mono text-xs`) and more spacious monospace treatments by editing a single setting.
- Field-label spans and config toggles in these key/value rows avoid hardcoded `text-xs` overrides for panel-specific traversal and 3D settings (for example, `graphRagWorkflow.maxHops` in GraphRAG workflow indexing, `three.layerOpacityByLayer[...]` and `schema.three.nodeSizingFormula` / `schema.three.edgeWidthFormula` in AI-KG Layers, `traversalStartNodeId` / `traversalMaxDepth` / `traversalLabelFilter` in Render presets, and the Arrow Length / Link Opacity / Node Motion / Fog / Starfield / Camera / Selection sliders in `ThreeViewTuningSection`). Instead, label Tooltips and select shells inherit `uiPanelKeyValueTextSizeClass` from `KeyTypeValueRow`, while chips, pills, and toolbar-style buttons remain free to opt into smaller `text-[10px]` sizing when they act as dense badges or controls (`canvas/src/features/panels/views/GraphRagWorkflowIndexingSection.tsx`, `canvas/src/features/panels/views/AiKgLayersSection.tsx`, `canvas/src/features/panels/views/RenderPresetSection.tsx`, `canvas/src/features/panels/views/ThreeViewTuningSection.tsx`).
- Micro-label typography uses a panel-scoped alias for helper text while preserving icon density controls. Schema/property badges in hover tooltips, the Node editor, and the Advanced schema editor continue to use `getBadgeChipClass` with `uiIconBadgeChipClass` and `uiIconBadgeChipTextSizeClass` so inline property badges and small schema tokens stay visually aligned with icon badge chips when teams tighten or loosen the “UI Density: Icons” badge text size. Panel helper text (for example, Advanced schema editor serialization error messages, JSON editor validation errors, parser/schema/code error banners, and hover tooltip property descriptions) reads `uiPanelMicroLabelTextSizeClass` from the shared UI slice; when this value is empty, components fall back to either `uiIconBadgeChipTextSizeClass` (for badge-adjacent tooltips) or their previous text size (for legacy `text-xs` helpers), so micro helper text tracks badge chip density or prior sizing by default but can be tuned independently under “UI Density: Panels” when needed (`canvas/src/hooks/useGraphStore.ts`, `canvas/src/hooks/store/uiSlice.ts`, `canvas/src/features/schema-editor/AdvancedSection.tsx`, `canvas/src/components/GraphHoverTooltip.tsx`, `canvas/src/components/NodeEditor.tsx`, `canvas/src/features/json/JsonEditor.tsx`, `canvas/src/components/BottomPanel/BottomPanelParserTab.tsx`, `canvas/src/components/BottomPanel/BottomPanelSchemaTab.tsx`, `canvas/src/components/BottomPanel/BottomPanelCodeTab.tsx`, `canvas/src/features/schema/ui/SchemaUiEditorRows.tsx`, `canvas/src/features/panels/views/ParserSchemaTabContent.tsx`, `canvas/src/features/settings/registry-ui.ts`, `canvas/src/features/panels/views/SettingsView.tsx`).

## Selection Semantics (2D & 3D)
- Selection state (`selectedNodeId`, `selectedEdgeId`, `selectionSource`) lives in `selectionSlice` and is shared by 2D and 3D renderers (`canvas/src/hooks/store/selectionSlice.ts`, `canvas/src/hooks/useGraphStore.ts`).
- 2D SVG (`GraphCanvas.tsx`) and 3D R3F scene (`ThreeGraph.tsx`) apply consistent highlighting rules: selected node/edge and its neighbors are emphasized; all non‑related nodes/edges are softly dimmed instead of removed.
- 3D uses the same selection source (`canvas` vs `panel`) so clicks in either mode update the common selection model and keep UI affordances aligned.
- The AI‑KG workflow reuses this selection model to drive a minimal traversal animation: the bottom panel **Render** tab exposes an `AI KG Layers & Traversal` block with sliders for `Layer 1/2/3 Opacity`, `Force Separation`, and `Traversal Delay (ms)`. The `AI KG Traversal` button in this block steps through edges defined by `graphRAGPath.traverse` in the data; `findGraphRagTraversalEdgeIds` derives the edge sequence and the button selects each edge in turn, causing the shared selection visuals to highlight the “active” part of the graph without introducing a separate animation subsystem. Layer sliders write to `schema.three.layerOpacityByLayer`, the separation slider wraps `setCharge` over `layout.forces.charge`, and the delay slider controls pacing only in the Render tab, keeping core selection/physics APIs unchanged (`canvas/src/lib/graph/graphragTraversal.ts:1–49`, `canvas/src/features/panels/views/RenderSettingsSection.tsx`, `canvas/src/features/three/ThreeGraph.tsx:215–297`, `canvas/src/components/GraphCanvas.tsx`).

## Launch Spotlight Card & Config
- Config‑driven steps: `SPOTLIGHT_STEPS` in `canvas/src/features/spotlight/config.ts` stores the four Launch spotlight steps with `id`, `title`, `body`, `targetSelector`, `variant: 'primary' | 'secondary'`, and optional `requires` keys (`'parserOpen'`, `'renderOpen'`, `'traversalRan'`). Copy is sourced from `WORKFLOW_STEP_COPY` and `getWorkflowStepBody` so the Help panel Workflow section and the spotlight card share the same language (`canvas/src/features/panels/config.ts`).
- Layout and anchoring: `LaunchSpotlight` renders a draggable card overlay anchored either near the tab or control indicated by `targetSelector` or, when no anchor is found, in the top‑right fallback position. The overlay uses a fixed, pointer‑gated container (`fixed inset-0` with `pointer-events-none` / `pointer-events-auto`) and `useSpotlightAnchor` to position the card above the Parser, Schema, and Render affordances (`canvas/src/features/spotlight/useSpotlightAnchor.ts`, `canvas/src/pages/Canvas.tsx:111–137`).
 - Positioning knobs: the Panel Settings tab exposes `spotlight.margin` and `spotlight.nearTopThreshold` as numeric settings that write into `spotlightMargin` and `spotlightNearTopThreshold` in the shared store (`canvas/src/hooks/useGraphStore.ts:14–79,192–201`). `getSpotlightCardStyle` reads these values to control viewport clamping margin and the “near top of viewport” threshold for flipping an anchored card below its target (`canvas/src/features/spotlight/positioning.ts:18–63`). Defaults are 8px margin and a 96px near‑top threshold.
- Minimize and replay: the card exposes `Minimize`, `Dismiss`, and `Next` / `Done` actions, plus a minimized pill with `Dismiss` / `Reopen` buttons so the guided tour behaves like an embed that can be tucked away but resumed. Visibility is controlled by the toolbar `Launch` button, Panel Help `Launch`, the Panel Settings `enableLaunchSpotlight` toggle, and the Cmd/Ctrl+Shift+G shortcut (`canvas/src/pages/Canvas.tsx:31–160`, `canvas/src/components/IconButton.tsx`).
- Requirements and gating: `LaunchSpotlight` computes a `requirementStatus` map from store state (Parser/Render bottom panel tabs and `aiKgTraversalRan`) and only enables `Next` when all `requires` for the current step are satisfied, showing short inline helper text when gating is active (for example, prompting the user to open Render or run AI‑KG Traversal). This replaces earlier index‑based conditions and keeps the tour extensible and embed‑friendly (`canvas/src/pages/Canvas.tsx:72–110`).

### Panels UI Layout Patterns

- **Small key/value rows (`KeyValueRow`)**
  - Purpose: compact “label → control” rows for settings and presets across MainPanel, BottomPanel, and floating Graph Traversal panels.
  - Layout primitive: `canvas/src/features/panels/ui/KeyValueRow.tsx`.
  - Typical usage:

    ```tsx
    <KeyValueRow
      label={<span className="text-xs text-gray-700">Traversal Delay (ms)</span>}
      control={(
        <>
          <input
            type="range"
            min={300}
            max={2000}
            step={100}
            value={traversalDelayMs}
            onChange={e => setTraversalDelayMs(Number(e.target.value))}
          />
          <span className="text-xs text-gray-600">{String(traversalDelayMs)}</span>
        </>
      )}
    />
    ```

  - Density: `density="default"` for most rows; `density="compact"` for dense multi-row grids (for example, GraphRAG indexing sliders/inputs).

- **Dense multi-row grids of key/value rows**
  - Purpose: lay out many related `KeyValueRow` entries in 2-column grids without changing the row primitive.
  - Layout pattern: local `div` wrappers with `grid grid-cols-2 gap-*` around multiple `KeyValueRow` instances.
  - Typical usage:

    ```tsx
    <div className="grid grid-cols-2 gap-3">
      <KeyValueRow /* row 1 */ />
      <KeyValueRow /* row 2 */ />
      <KeyValueRow /* row 3 */ />
      <KeyValueRow /* row 4 */ />
    </div>
    ```

  - Scope: used in BottomPanel Orchestrator/Render sections, AI-KG layers controls, and 3D tuning panels where many small rows must fit in a narrow column.

- **Paired editors and admin panels (`TwoColumnEditorGrid`)**
  - Purpose: two peer editors or admin forms side by side (for example, Template vs Properties, Validation vs Rules, node types vs edge labels).
  - Layout primitive: `canvas/src/features/panels/ui/TwoColumnEditorGrid.tsx`.
  - Typical usage:

    ```tsx
    <TwoColumnEditorGrid className="mt-2">
      <div className="flex min-h-0 flex-col">
        <SchemaSubstepHeader title="Template" />
        <JsonEditor
          value={templateText}
          onChange={setTemplateText}
          className="w-full flex-1 min-h-0"
          language="json"
        />
      </div>
      <div className="flex min-h-0 flex-col">
        <SchemaSubstepHeader title="Properties" />
        <JsonEditor
          value={propsText}
          onChange={setPropsText}
          className="w-full flex-1 min-h-0"
          language="json"
        />
      </div>
    </TwoColumnEditorGrid>
    ```

  - Scope: reserved for “two big peers” (paired editors or admin panels) in bottom-panel schema UI editors, advanced schema admin panels, and any future main or floating editor panels. Small key/value rows and dense multi-row grids continue to use `KeyValueRow` plus local layout wrappers.

### UI copy single‑sourcing

- Shared UI strings that appear in multiple panels or controls are centralized in `UI_LABELS` (`canvas/src/lib/config.ts`) so layout‑related copy, action labels (Apply, Close, Undo, Redo, Restore), and affordance text stay consistent across the Canvas.
- Tooltip and handle copy for panel resizing, such as the bottom panel and collapsible panel resize handles, should reference `UI_LABELS` entries (for example, `UI_LABELS.dragToResize`) instead of hard‑coded strings when adding new resize or layout affordances.
