### AI‑KG Layer Modes Details
- Modes: `schema.layers.mode` supports `'property'`, `'document-structure'`, and `'semantic'`. The active mode is stored on `GraphSchema.layers` and can be toggled from the toolbar **Layer Mode** button, the Render panel’s **AI‑KG Layers** section (including the floating renderer view), and the Graph Fields panel header.
- Styling: all three modes share the same node and edge visual helpers (`getRenderNodeRadius2d`, `getEdgeStrokeWidth`, and `schema.three.{nodeSizingFormula,edgeWidthFormula}`), so **property**, **document‑structure**, and **semantic** modes use the same schema‑driven node/edge color palette and base edge widths. The modes differ by wiring and layering behavior: semantic mode layers on top of the shared styling by sizing nodes from `visual:importance` and scaling edge widths from `visual:weight` when similarity edges are active; the detailed node, edge, and layer color semantics for this overlay follow the neutral rules described in the separate **Semantic layer mode and controls** document (`knowgrph-semantic-mode-document.md`) and its AgenticRAG JSON‑LD directive (`schema/AgenticRAG/semantic-mode.jsonld`).
- Semantic polygons vs. JSON‑LD blocks: in **semantic** mode, polygon layers represent semantic communities (derived from similarity edges) rather than JSON‑LD structural block types. JSON‑LD block types such as `Document`, `Section`, `Paragraph`, `CodeBlock`, `Table`, `List`, and `ListItem` are excluded from semantic community polygon grouping so “word-frequency / meaning” clusters remain the primary polygon surface. Separately, the shared **Canvas Group Polygons** feature can still draw hulls from array-valued node references (for example `steps`/`contains`) when enabled, regardless of layer mode. In **document‑structure** mode, structural block types drive non‑polygon **type bands** by mapping node types to `node.properties['visual:layer']` (for example `Document`/`Section` to layer `3`, `Paragraph`/`Table`/`List` to layer `2`, and `CodeBlock`/`ListItem` to layer `1`); the 3D renderer reads `three.layerOpacityByLayer` to fade foreground vs. mid vs. background bands while keeping node shapes and colors consistent with the other modes.
- Semantic similarity overlay: semantic mode builds a token‑based similarity graph over configured text fields using `schema.layers.semantic.{similarityMetric,similarityEdgeLabel,topKEdgesPerNode,minSimilarity}`. The derived similarity edges are rendered on the same canvas as the document‑structure‑driven bands, so users can see which nodes are textually related inside and across document sections without changing datasets or presets.
- Codebase path provenance and camera anchoring: when a Graph Data Table row includes a `codebasePath` value, the table exposes it as plain text backed by `metadata.codebasePath` so curators can scan and filter provenance deep-links alongside traversal context. Canvas keeps the 2D/3D camera aligned with the current graph selection as users switch tabs and panels, but it no longer opens external files inside the bottom panel editor. Repository-relative `codebasePath` values remain usable by external tools and dev-server helpers that resolve real files via `/@fs` URLs with `VITE_CODEBASE_ROOT`, while the canvas UI focuses on graph structure, traversal overlays, and layer modes.
- AgenticRAG schema alignment: the AgenticRAG JSON‑LD schemas (for example `schema/AgenticRAG/semantic-mode.jsonld` in the separate AgenticRAG catalog) mirror the same layer configuration fields (`schema.layers.mode`, `schema.layers.semantic.*`, and `schema.metadata["canvas:polygons"]`). Graphs produced by these workflows load into the canvas with the same layer modes, polygon styling, and similarity settings already encoded, so semantic overlays behave identically between offline indexing and in‑app visualization.

When this graph is loaded into Canvas with the default schema:

- **Property mode** groups nodes by array‑valued properties such as `tags`. With the Lean Startup MVP palette, tags like `idea`, `hypothesis`, and `execution` map onto the blue/yellow/green node colors described by `MVP_COLOR_PALETTE.nodes`, so category bands and traversal overlays share the same semantic colors across views.
- **Document‑structure mode** groups by node type (`Document`, `Section`, `Paragraph`, `CodeBlock`) while keeping these structural nodes visible as circles. Layer bands derive from type‑based `visual:layer` assignments and `schema.layers.documentStructure.minGroupSize`, while node colors remain controlled by the shared palette (for example mapping `Document` to the idea color, `Section` to the hypothesis color, and execution‑oriented blocks to the execution color).
- **Semantic mode** hides the structural container types as individual circles and instead derives convex‑hull polygons from a combination of JSON‑LD array‑based groups (for example `contains`), document‑structure block groups, and semantic communities built from similarity edges. Tokenization over `chunk_text` builds a similarity graph whose strongest edges are labeled with `schema.layers.semantic.similarityEdgeLabel`; derived importance and community fields are attached as `visual:*` metadata so semantic overlays can highlight per‑document bands and global semantic neighborhoods without altering the underlying Lean Startup MVP palette.

### Quick Reference
- 3D defaults: `schema.three.linkDirectionalArrowLength: 8`, `linkOpacity: 0.55`, `linkCurvature: 0.0`, `linkCurveRotation: 0.0`, `linkDirectionalArrowRelPos: 0.85`, `linkDirectionalParticles: 0`, `linkDirectionalParticleSpeed: 0.6`, `sphereRadius: 120`, `seed: 1`, `minSpacing: 0`, `nodeMotionIntensity: 1.0`, `minimapOpacity: 0.7`, `selection.selectedNodeGlowIntensity: 0.8`, `selection.dimmedNodeOpacity: 0.2`, `selection.dimmedEdgeOpacity: 0.2`, `selection.selectedEdgeWidth: 3`, `layerOpacityByLayer:{'1':1.0,'2':0.9,'3':0.8}` (`canvas/src/lib/graph/schema.ts`).
- Link distances: honors `layout.forces.linkDistanceByLabel[label]` when present (`canvas/src/components/GraphCanvas/simulation.ts:23`, 3D spring in `Physics3D`).
- Selection: 3D highlights nodes/edges using the same selection state as 2D (selected node/edge and neighbors) via `useGraphStore.selectedNodeId/selectedEdgeId`; non‑selected elements are dimmed rather than removed. Selected nodes use emissive glow; dimming and selected edge widths are driven by `schema.three.selection.*` and centralized helpers `getThreeConfig` / `getThreeSelectionConfig` (`canvas/src/lib/graph/schema.ts:254–289`, `canvas/src/components/GraphCanvas.tsx`, `canvas/src/features/three/selection.ts`, `canvas/src/features/three/ThreeGraph.tsx`).
- Selection‑aware D3 charts: Graph Data Table aggregates and Dataset Inspector mini‑visualizations run over either the full dataset or the current selection neighborhood. The Graph Data Table uses its **Selection view** toggle together with `selectedNodeId` / `selectedEdgeId` and the multi‑select arrays `selectedNodeIds` / `selectedEdgeIds` to derive a selection subgraph via shared helpers (for example, the `SelectionAnchorIds` alias normalized by `normalizeSelectionIds` and passed to `buildSelectionSubgraphForAnchorIds` in curator state), and Dataset Inspector exposes an `Auto / Dataset / Selection` toggle. Both pathways treat the selection neighborhood as a multi‑node/multi‑edge subset, so radial/cluster, polygon, and path‑style charts stay aligned with whatever is currently selected on the canvas without hard‑coding dataset‑specific filters.
- Canvas group polygons metadata: `schema.metadata["canvas:polygons"]` supplies `defaultStyle`, `byOwnerType`, and `byPropertyKey` style maps plus optional `groupingLogic`, `layer`, `label`, `tooltip`, and `schemaDrivenStylingEntrypoint` fields used as metadata hints; these defaults are editable from the Main Panel **Graph Fields** tab via the polygon presets editor in Field Settings.
- Canvas zoom toggles and overlay: toolbar Fit to Screen (`fitToScreenMode`) and Zoom to Selection (`zoomToSelectionMode`) are persistent, mutually exclusive zoom modes that reuse shared `zoomRequest` / `threeCameraRequest('selection')` handlers and selection state so 2D/3D views stay centered on either the full dataset or the active selection. Embed/Overlay (`embedOverlayMode`) layers mini‑visualizations on top of whichever region is in focus.
- Panel → Render groups 2D/3D tuning into a **Render / Inspect (Renderer, indexers)** section plus supporting tools. Render / Inspect combines schema-driven presets with live tuning controls: presets are applied explicitly from the UI (no dataset-name detection) and are kept domain‑agnostic by reading node/edge types from `schema.catalog.*`, while the 3D tuning surface exposes collapsible groups for **Edges and particles**, **Layout and geometry**, **Background and fog**, **Starfield**, **Camera and navigation**, and **Selection highlighting** implemented via `RenderSettingsSection`, `RenderPresetSection`, the feature-scoped `renderPresetCatalog` preset registry, `ThreeViewTuningSection`, and the `ThreeView*Section` files (`canvas/src/features/panels/views/RenderSettingsSection.tsx`, `canvas/src/features/panels/views/RenderPresetSection.tsx`, `canvas/src/features/panels/views/renderPresetCatalog.ts`, `canvas/src/features/panels/views/ThreeViewTuningSection.tsx`, `canvas/src/features/panels/views/ThreeViewLinksSection.tsx`, `canvas/src/features/panels/views/ThreeViewLayoutSection.tsx`, `canvas/src/features/panels/views/ThreeViewBackgroundFogSection.tsx`, `canvas/src/features/panels/views/ThreeViewStarfieldSection.tsx`, `canvas/src/features/panels/views/ThreeViewCameraSection.tsx`, `canvas/src/features/panels/views/ThreeViewSelectionSection.tsx`). Dataset mini‑visualizations live under **Dataset Inspector**, and the **Codebase index pipeline** section simply exposes the AgenticRAG-aware indexing command without adding new renderer settings.
