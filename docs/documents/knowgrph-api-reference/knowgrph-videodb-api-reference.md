## Async vs Synchronous Operation Notes

VideoDB uses a shared async response pattern. Operations that require significant processing time return a job identifier immediately; the caller must poll `GET /async-response/{id}` until the job reaches a terminal state.

### Synchronous operations (response is the result)

These endpoints return their result directly in the HTTP response with no polling required:

- `GET /collection` â€” list all collections
- `POST /collection` â€” create a collection
- `GET /collection/{id}` â€” get a collection by id
- `DELETE /collection/{id}` â€” delete a collection
- `GET /collection/{id}/video` â€” list videos in a collection
- `GET /video/{id}` â€” get a video by id
- `DELETE /video/{id}` â€” delete a video
- `PUT /video/{id}` â€” update a video
- `POST /video/{id}/stream/` â€” get a stream URL (synchronous; returns URL immediately)
- `GET /video/{id}/transcription` â€” retrieve completed transcription
- `GET /health` â€” health check / readiness probe
- `POST /chat` â€” conversational query
- `GET /async-response/{id}` â€” the polling endpoint itself responds synchronously

### Async operations (returns a job id; poll to completion)

These endpoints return a job identifier. The caller must poll `GET /async-response/{id}` to retrieve the result:

- `POST /collection/{id}/upload` â€” upload a video to a collection
- `POST /video/{id}/transcription` â€” create a transcription
- `POST /video/{id}/index/` â€” create a spoken-word index
- `POST /video/{id}/index/scene/` â€” create a scene index
- `POST /generate_video` (AI generation) â€” generate video
- `POST /generate_audio` (AI generation) â€” generate audio
- `POST /generate_image` (AI generation) â€” generate image
- `POST /generate_text` (AI generation) â€” generate text
- `POST /dub_video` (AI generation) â€” dub a video
- `POST /translate_video` (AI generation) â€” translate a video
- `POST /timeline/compile` â€” compile a timeline into a stream
- `POST /download` â€” create a download job
- `POST /rtstream/{id}/status` (start) â€” start an RTStream
- `POST /rtstream/{id}/status` (stop) â€” stop an RTStream

### Polling circuit-breaker

The shared polling endpoint is `GET /async-response/{id}`. Terminal states are `completed` and `failed`.

**Maximum polling bound: 36 iterations at 10-second intervals** (~6 minutes total), matching the Gemini Veo harness pattern in `canvas/src/features/chat/geminiRunGeneration.ts`. Polling **MUST stop and return a failure result** when this bound is reached â€” it must never loop indefinitely. The `videodb.async_response.get` row documents 60 as the absolute maximum poll cap; 36 is the firm operational bound aligned to the Gemini Veo pattern.

## Pipeline Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system â€” essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Upload â†’ Index â†’ Search Pipeline Completeness

*For any* uploaded video whose transcript contains at least one word matching a given query, calling `POST /video/{id}/index/` (spoken-word index) followed by `POST /video/{id}/search/` with `index_type: spoken_word` and that query SHALL return a result set containing at least one item.

**Validates: Requirements 18.2**

Classification: **Suitable for property-based testing with mocked HTTP layer.** The VideoDB HTTP layer must be mocked to keep tests cost-free. Generate random video IDs, transcript content, and query words; mock the upload, index, and search endpoints; verify the result set is non-empty when the query word appears in the transcript.

### Property 2: Stream URL Validity

*For any* valid video ID, after a successful `POST /video/{id}/stream/` call, the returned stream URL SHALL be a non-empty string beginning with `https://` or `http://`.

**Validates: Requirements 18.3**

Classification: **Suitable for property-based testing with mocked HTTP layer.** Mock the stream endpoint returning various URL shapes; verify the knowgrph response parser only accepts and forwards URLs that begin with `https://` or `http://` and are non-empty.

### Property 2A: Character Clip Timeline Completeness

*For any* approved subject timeline packet, each subject SHALL contain at least one valid `[start, end]` range where `end > start`, and a VideoDB SDK call to `video.generate_stream(timeline=...)` SHALL only write a clip URL after a live response returns a non-empty `http://` or `https://` value.

**Validates: Requirements 18.3A**

Classification: **Suitable for property-based testing with mocked SDK layer.** Mock `generate_stream(timeline=...)`; generate subject timeline ranges from imported metadata, search results, or operator-approved Strybldr cards; verify blank clips stay blank until a live SDK response returns a valid URL.

### Property 3: Transcription Round-Trip

*For any* video with audible speech, after `POST /video/{id}/transcription` completes polling and reaches terminal state `completed`, calling `GET /video/{id}/transcription` SHALL return a transcription object whose text content is non-empty.

**Validates: Requirements 18.4**

Classification: **Suitable for property-based testing with mocked HTTP layer.** Mock the async transcription creation, polling loop, and retrieval. Generate random video IDs and non-empty transcript payloads; verify that the round-trip (create â†’ poll to terminal â†’ get) preserves a non-empty text field.

### Property 4: Async Polling Termination

*For any* async job ID polled via `GET /async-response/{id}`, the polling harness SHALL call the endpoint at most 36 times and SHALL return a failure result when the bound is reached, regardless of whether the job has reached a terminal state.

**Validates: Requirements 18.5**

Classification: **Suitable for property-based testing with mocked HTTP layer.** Mock the polling endpoint to always return a non-terminal state; verify the harness terminates after exactly 36 iterations and returns a failure result rather than looping indefinitely. This is a pure function test on the circuit-breaker logic.
