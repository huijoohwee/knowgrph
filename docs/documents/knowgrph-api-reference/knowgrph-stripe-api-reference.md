## knowgrph Payment Worker Overlay

The generated Stripe rows below remain the Stripe OpenAPI source map. knowgrph accepts Stripe payments only through the shared payment Worker and shared payment contracts, so local readiness is controlled by the following source-owned boundaries:

| area | source | acceptance boundary |
| --- | --- | --- |
| Worker credentials | `grph-shared/src/payments/stripePaymentSsot.ts`; `scripts/{configure-stripe-payment-worker,check-stripe-payment-readiness}.mjs`; `cloudflare/workers/knowgrph-payment/payments.ts` | `STRIPE_RESTRICTED_KEY` or `STRIPE_SECRET_KEY`, plus `STRIPE_WEBHOOK_SECRET`, are Cloudflare Worker secrets only. The configure/readiness helpers fail if these names appear in visible Worker `[vars]`. |
| Existing Worker secrets | `scripts/configure-stripe-payment-worker.mjs`; Wrangler `secret list` | Existing `STRIPE_SECRET_KEY`/`STRIPE_WEBHOOK_SECRET` Worker secret names satisfy configure checks when the operator is only writing visible checkout price authority. |
| Checkout authority | `cloudflare/workers/knowgrph-payment/payments.ts`; `grph-shared/src/payments/stripePaymentSsot.ts`; `scripts/configure-stripe-payment-worker.mjs` | Checkout price authority, `STRIPE_CHECKOUT_MODE`, and `STRIPE_CHECKOUT_RETURN_ORIGIN` are visible non-secret Worker `[vars]`, not Worker secrets. Operator-supplied checkout price authority is written to `wrangler.toml` only with `--write-visible-vars --yes --confirm=apply-stripe-payment-worker-config`; deploy `payment:worker:deploy` or use explicit `--deploy-visible-vars --apply --yes --confirm=apply-stripe-payment-worker-config` before live Checkout smoke. Return URLs are validated against the configured return origin when present, otherwise the Worker route origin. |
| Hosted Checkout return | `canvas/src/features/payments/{stripeCheckout,StripeCheckoutReturnRuntime}.tsx`; `cloudflare/workers/knowgrph-payment/payments.ts` | `success_url` carries `session_id={CHECKOUT_SESSION_ID}` for hosted Checkout returns. Public status lookups accept only canonical `session_id`, read the local D1 Checkout row first, and never expose customer identifiers, Stripe metadata, hosted Checkout URLs, or workspace ids. |
| Agentic reconciliation | `cloudflare/workers/knowgrph-payment/{agenticCommerce,agenticCommerceSettlement,payments}.ts`; `grph-shared/src/payments/agenticCommerceSsot.ts` | Agentic Checkout stores the ACP session id in Stripe `metadata[acp_session_id]`, `client_reference_id`, and Stripe `Idempotency-Key`, then settles only when payment status, ACP id, amount, and currency match local D1 state. |
| Machine-native x402 | `cloudflare/workers/knowgrph-payment/agenticCommerceX402.ts`; `scripts/{agent-ready-commerce-checks,check-agent-ready-commerce,configure-x402-payment-worker}.mjs`; `grph-shared/src/payments/agenticCommerceSsot.ts` | `GET /api`, `GET /api/v1`, and `GET /api/payments/commerce/x402` are Worker-owned x402 paid-resource probes. `npm run payment:x402:configure` writes operator-owned `X402_PAY_TO_ADDRESS` to `knowgrph-payment` Worker `[vars]` only with `--write-visible-vars --yes --confirm=apply-stripe-payment-worker-config`; `npm run payment:x402:readiness` verifies ACP, UCP, MPP, and HTTP 402 `payment-required` metadata, and fails if the `payTo` address is the deterministic fallback. |
| Combined payment readiness | `scripts/check-payment-readiness.mjs`; `scripts/{check-stripe-payment-readiness,check-agent-ready-commerce}.mjs` | `npm run payment:readiness -- --live-checkout-create` aggregates the focused Stripe and x402 readiness gates after visible payment authority is configured and deployed; the wrapper does not own price, secret, D1, or x402 semantics. |
| D1 webhook processing | `cloudflare/d1/migrations/0006_stripe_webhook_processing_state.sql`; `cloudflare/workers/knowgrph-payment/payments.ts` | Remote D1 must include payment tables plus `stripe_webhook_events.processing_status` and `processing_error`; `processed_at` and `processing_error` stay nullable so in-flight and failed Stripe webhook claims are retryable instead of frozen. |
| Canvas paywall | `canvas/src/features/payments/{PaywallOverlay,stripeCheckout,StripeCheckoutReturnRuntime}.tsx` | Each Open Checkout action creates a fresh server-managed hosted Session and redirects the current browser window. The hosted Checkout URL stays transient runtime state for redirect only; it is not persisted and is not rendered back into the overlay. |
