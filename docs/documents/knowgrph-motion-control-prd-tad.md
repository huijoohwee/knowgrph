# Knowgrph Motion Control PRD/TAD

## Product contract

Motion Control is a first-class FloatingPanel view between Animation and Camera. It reuses the existing FloatingPanel catalog header, surface/body layout, form controls, local status rows, and Markdown invocation renderer. Entering Toolbar → Canvas View Mode → Surface Mode → XR opens Motion Control only when no FloatingPanel is already open and keeps BottomPanel Timeline as the sole transport.

Two compact target cards project the current **3D for XR** and **Animation** state into Motion Control. Their Open actions move the same FloatingPanel to Media's canonical 3D view or the canonical Animation view; they do not copy either catalog, create another target store, or introduce a second scene/animation mutation path. Each card renders the existing scene or animation WebMCP identity and its existing `/`, `@`, and `#` invocation rather than defining a Motion-Control-specific replacement.

The operator explicitly starts and stops one local camera session. A square mirrored preview shows the latest video frame and accepted skeleton; its **Bounding box** checkbox is disabled by default and can project the already-computed tracked pose ROI around the person. The same preference also renders one catalog-dimension outline around every scene-authored XR subject in both the ordinary XR stage and native-controller branch. Those outlines identify existing subject IDs, labels, categories, and assets; they are not camera detections, track identities, or biometric identities. The checkbox is a page-session preference that survives Stop, restart, and panel remount, while camera-derived coordinates remain transient and clear with tracking or runtime loss. This projection adds no second detector, inference pass, visual-annotation runtime, or external dependency. Status rows report permission, requested and effective backend, fallback reason, confidence, inference latency, and effective FPS. The panel instructs the operator to keep one full body centered because the MVP runs Google's standalone landmark model with a bounded centered-person crop rather than claiming arbitrary-frame person detection or multi-person tracking.

The same view now owns a provider-neutral capture session. Browser video, depth, landmark-stream, and explicitly shared peer-derived providers register through one finite contract with locally generated opaque IDs. Capture-time timestamps, clock alignment, calibration provenance, synchronization skew, jitter, sequence loss, missing samples, confidence, landmark visibility/presence, and bounded recording evidence produce the honest tiers `single-view-control`, `time-aligned-multi-source`, and `calibrated-metric-reconstruction`. The built-in monocular source never becomes research-ready by default. Research readiness requires at least two synchronized metric sources with canonical session-clock or measured source-local alignment, measured calibration provenance, monotonic sequence evidence, minimum confidence and landmark-evidence ratios, and bounded quality. Skills & Commands and Media project this same session and canonical invocation/runtime; they do not create another capture store.

## Agentic ECS composition boundary

Agentic ECS remains a numeric world/decision system, not a game engine, renderer, animation renderer, browser-scene owner, or physics owner. Its private local-stdio lane exposes exactly `knowgrph.ecs.session_start` (`/ecs.session-start #agentic-ecs @source.frontmatter`), `knowgrph.ecs.world_tick` (`/ecs.world-tick #agentic-ecs @ecs-session`), and `knowgrph.ecs.decision_persist` (`/ecs.decision-persist #agentic-ecs @ecs-session`). That session lane is intentionally disconnected from browser XR scene, Animation, Motion Control, and physics state and adds no browser WebMCP surface.

The honest shared chain is composition: 3D asset selection and `kgXrMotionReference` materialization remain with the XR scene owner; authored Animation and Motion Control remain with their existing browser WebMCP and `/ @ #` contracts; Agentic ECS may project a caller-held World through `agenticEcsCanvasProjection.ts` into the existing `applyChatKgcDocumentTextToCanvas` KGC/Canvas apply seam. This feature adds no XR-to-ECS entity bridge and makes no live-ECS-control claim. A live claim would require separate proof of an actual stdio `session_start` → `world_tick` → `decision_persist` sequence and an explicitly approved browser-state adapter.

## Official LiteRT and model boundary

- Runtime dependency: [`@litertjs/core`](https://www.npmjs.com/package/@litertjs/core), maintained by [Google AI Edge LiteRT](https://github.com/google-ai-edge/LiteRT), Apache-2.0.
- Runtime loading: the application serves the package's Wasm artifacts from its own origin, calls `loadLiteRt`, compiles the model with `loadAndCompile`, awaits inference, and explicitly deletes input tensors, output tensors, and compiled models.
- Model source: Google's official [Pose Landmarker Full task](https://storage.googleapis.com/mediapipe-models/pose_landmarker/pose_landmarker_full/float16/latest/pose_landmarker_full.task), from which the build step extracts only `pose_landmarks_detector.tflite`.
- Task SHA-256: `4eaa5eb7a98365221087693fcc286334cf0858e2eb6e15b506aa4a7ecdcec4ad`.
- Extracted model SHA-256: `82be6d591b9dad7d29fe21dc9fd892bf8b9602c458fb05209283de8282a0c488`.
- Expected input: float32 `[1,256,256,3]`, RGB normalized to `[0,1]`.
- Expected public outputs: 33 normalized landmarks, 33 hip-relative world landmarks, and pose confidence. Output roles are resolved and shape-checked from model metadata rather than silently assuming output order.

The runtime prefers WebGPU when requested and supported. It reports pure `webgpu` only when LiteRT confirms full delegation, `webgpu+wasm` for partitioned execution, and `wasm` when capability, delegation, or compilation falls back to CPU. `auto`, `webgpu`, and `wasm` are preferences, not proof of the effective backend; inspection reports both requested and effective values.

## Capture, scheduling, and privacy

1. Camera access is requested only from explicit Start control after XR activation succeeds; a document lock or other rejected XR transition fails before permission or capture begins.
2. The first input is a centered square person crop; accepted landmarks produce a bounded next-frame ROI. When the operator enables Bounding box, that same ROI is mirrored into the preview without a second landmark scan or detector. Tracking loss resets to the centered acquisition crop and clears the transient box coordinates.
3. Only one inference may be active. The next animation frame is scheduled after the current asynchronous inference settles, so slow inference cannot queue frames.
4. Invalid shapes, non-finite values, and pose confidence below `0.5` clear the live pose and transient preview box instead of freezing stale motion.
5. Capture remains active across the open Motion Control, Skills & Commands, Media, Animation, and Game Mode FloatingPanel surfaces only while XR remains active. Explicit Stop, closing the FloatingPanel, selecting an unrelated view, leaving XR, page hide, camera end, backend replacement, lifecycle-owner unmount, or runtime error cancels scheduling; stops every `MediaStreamTrack`; detaches the video; clears canvases, pose, and transient box coordinates; deletes LiteRT resources; and disables peer sharing. The page-session Bounding box preference remains unchanged across Stop and restart.
6. A valid live pose transiently overrides only the selected compatible humanoid's character pose. Authored animation assignments and action-path marks stay canonical and resume when tracking is lost or Motion Control stops.
7. Camera frames, raw tensors, preview-box coordinates, stable device/peer identities, and endpoints are not uploaded, persisted to graph/workspace storage, included in exports, or returned through MCP. Recording is explicit, browser-local, memory-bounded, and limited to derived landmarks plus quality/evidence metadata. Finish or Stop freezes the bounded recording for deterministic local export; Clear releases it. WebMCP returns export metadata, never content bytes.

The MVP is single-person entertainment/XR control. It does not claim identity recognition, surveillance, multi-person tracking, medically meaningful biomechanics, metric-accurate camera depth, or safety-critical control.

## Clean-room inspiration boundary

[andrisgauracs/LiteRT.js-Mocap](https://github.com/andrisgauracs/LiteRT.js-Mocap) was consulted only for neutral product-level ideas: local browser inference, backend selection, ROI continuity, smoothing, confidence-aware pose consumption, and XR/avatar control. The repository is not a dependency or implementation source.

[FreeMoCap](https://github.com/freemocap/freemocap) and its official calibration, triangulation, and camera-system documentation were consulted only for neutral principles: accessible capture, separable capture/processing responsibilities, multi-source time alignment, explicit calibration, evidence-aware reconstruction quality, and portable research data. FreeMoCap is not a dependency, service, subprocess, model provider, or implementation/schema source.

Knowgrph must not copy or adapt either reference's code, algorithms' expression, file/module structure, comments, prose, configuration, data schemas, UI/CSS, tests, fixtures, assets, model binaries, screenshots, examples, or build layout. Runtime and build code must not import, fetch, embed, clone, bridge, or call either repository. The implementation is independently authored from existing Knowgrph owners plus official Google/browser specifications. A repository-wide relevant-text and dependency-manifest test enforces forbidden owner/repository markers with a narrow allowlist for attribution and its enforcement tests; this is a guardrail, not proof of authorship.

## Invocation and physics projection

- Opening: `/motion.control @canvas #pose operation=open` activates XR and opens the panel without requesting camera permission.
- Bounding-box preference: `/motion.control @canvas #pose operation=open boundingBox=true` enables the reused live-pose ROI projection plus scene-authored XR subject outlines, and `boundingBox=false` disables both. Both forms only open/confirm the XR surface and update the page-session preference; neither starts the camera.
- Starting: `/motion.control @canvas #pose operation=start backend=<auto|webgpu|wasm>` first verifies the approved XR surface, then starts the active browser-local capture/inference runtime and surfaces XR-activation or permission failure honestly.
- Stopping: `/motion.control @canvas #pose operation=stop` releases capture/inference and peer transport, removes transient local/peer sources, and finishes any active bounded recording for explicit Export or Clear.
- Recording: `operation=record` begins bounded derived-landmark capture, `operation=finish` freezes it, and `operation=clear` releases it. Recording requires at least one registered source.
- Export: `operation=export format=json|csv` builds the same deterministic local artifact used by Media; WebMCP returns digest, size, source/sample/landmark counts, and research-evidence state without content bytes.
- Peer sharing: `operation=share enabled=true|false` explicitly toggles derived-observation exchange over the active collaboration session. It cannot create an invitation, endpoint, or persisted document.
- WebMCP structured calls reuse the identical controller. `open` alone accepts optional `boundingBox`, `start` alone accepts `backend`, `export` requires `format`, and `share` requires `enabled`; other operations accept no auxiliary field. Wrong types, casing, duplicates, mixed invocation/structured fields, and unknown keys fail closed.
- Preview inspection returns whether the Bounding box preference is enabled and whether current tracking makes a camera ROI available. The separate 3D target projection may also return strict scene-authored subject metadata and physics attachment status, but never camera-box coordinates, landmarks, images, inferred camera identities, or history.
- The 3D for XR card reuses `knowgrph.control_local_xr_scene`, the existing XR scene invocation, shot-target selection, physics snapshot, and Simulation workbench; the Animation card reuses `knowgrph.control_local_animation` and the source-backed `/animation.control` grammar. Vehicle, prop, and other preset-compatible targets remain valid authored-animation targets even though only humanoids can receive live pose. `move-object` reports the constraint resolver's actual status, resolved position, and applied displacement. Motion Control adds no composite MCP tool, operation, invocation schema, or physics mutation path.
- The accepted pose projects transiently into the selected compatible humanoid through the existing animation pose seam. It overrides the visible character pose only; authored preset/path state is neither rewritten nor cleared and resumes after Stop or tracking loss. In the physics playground, torso lean and arm direction map to the native normalized controller input, which is merged with keyboard/gamepad input before the existing single deterministic physics step.

No invocation accepts an arbitrary model/source URL, Wasm URL, camera-frame payload, landmark payload, stable device/peer ID, endpoint, invitation token, node id, document path, or validation share token.

## Acceptance and proof boundary

- Focused tests cover first-class panel routing and shared Motion Control/Skills & Commands/Media projections; strict provider registration; immutable coordinate space and runtime-owned receive time; canonical/measured alignment, measured calibration, and shared-reconstruction provenance; honest tier downgrades; synchronized skew, jitter, missing and sequence-loss evidence; bounded recording, in-flight export invalidation, recording-local synchronized-group thresholds, and deterministic JSON/CSV digests; strict peer message size/shape/rate/backpressure/priority-cleanup/relay-alias/topology behavior; redacted inspection/export results; `/ @ #` and WebMCP convergence; default-off Bounding box behavior; lifecycle cleanup; official LiteRT API markers; source asset integrity; selected-humanoid/physics projection; the disconnected Agentic ECS boundary; repository-wide clean-room marker/dependency enforcement; and external validation-input hardcode rejection.
- Build validation must run `prepare:litert-assets`, TypeScript, focused runtime tests, docs checks, hygiene, and `git diff --check`.
- Browser validation must inspect a source-backed XR route; verify Bounding box remains independent from Start; record, Finish, download JSON and CSV through Media, and Clear; switch among Motion Control, Skills & Commands, Media, Animation, and Game Mode without losing capture; confirm an unrelated view stops it; and, with two peers, verify explicit sharing, disconnect cleanup, opaque source projection, and no frame/network persistence.
- Source/build proof does not by itself prove camera permission, live pose quality, effective WebGPU execution, calibrated metric reconstruction, multi-device clock quality, Prod, or Cloudflare deployment. Dev browser observations must state hardware, permission, peer, and calibration limitations explicitly.

VCC: Given a source-backed XR document, when the operator starts Motion Control and registers one or more local/provider/peer-derived sources, then the panel reports only evidence earned by finite capture-time, alignment, calibration, synchronization, and quality records. One monocular or uncalibrated source remains `single-view-control`; `researchReady=true` requires at least two synchronized metric calibrated sources with measured provenance, explicit measured shared-reconstruction evidence binding those calibrations into one session frame, bounded loss/jitter/skew, and at least three disjoint synchronized groups inside one stable reconstruction/source cohort in the exported recording. Unrelated self-declared metric streams, pre-recording quality, and groups accumulated across changed cohorts never qualify. Motion Control, Skills & Commands, and Media project the same session, tool pair, and `/motion.control @canvas #pose` owner; switching among the approved XR views retains capture, while XR exit, panel close, an unrelated view, page/camera loss, or explicit Stop releases capture and peer transport. Record/Finish/Stop preserve only a bounded derived-landmark artifact, Media downloads deterministic JSON/CSV locally, Clear releases it and invalidates an in-flight export, and WebMCP exposes aggregate/digest metadata without frames, tensors, landmark arrays, stable identities, endpoints, reconstruction evidence content, or content bytes. The existing XR scene, animation, physics, target, and Agentic ECS ownership boundaries remain unchanged.
