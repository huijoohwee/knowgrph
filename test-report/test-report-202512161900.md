### Parser Usage and Simplification Advice

**Current split**

- Canvas / CI (TypeScript):
  - Uses a single JSON-LD parser (`parseJsonLd` in `canvas/src/lib/graph/jsonld.ts:17-118`) for all three file sets.
  - Parser is structural only, consistent with `docs/knowgrph-codebase-audit-catalog.md:31-43`:
    - Focuses on `GraphData` shape (nodes, edges, properties).
    - Ignores domain-specific semantics and property meanings.
- Python:
  - Offline normalization uses the unified CLI:
    - `python -m knowgrph_parser jsonld-universal` – structural JSON/JSON-LD → `GraphData`.
  - A0, Universal Lean Startup, and AI-KG all work with the same structural normalizer when the goal is `GraphData`.

**Why AI-KG has dedicated Python CLI tooling**

- Complexity and multi-target outputs:
  - AI-KG embeds layers, sequences, traversal metadata, and RAG-related fields that need to be mapped into multiple shapes:
    - Canvas `GraphData` for interactive visualization.
    - D3-style `nodes`/`links` for external HTML/JS demos.
    - Regression fixtures for validating schema and triples.
- Historical and tooling reasons:
  - AI-KG originated as a Python-first experiment where data wrangling, ontology merges, and layout hints were easier to implement and iterate on.
  - Some of the AI-KG transformations and exports (for example D3-specific shapes) are better suited to batch/CLI workflows rather than running in the browser.

**Should this be simplified?**

- For canvas and CI:
  - The current approach is already simple:
    - One JSON-LD parser (`parseJsonLd`) for all three sets.
    - Schema and validation logic in TypeScript only.
  - Recommendation:
    - Keep the TypeScript side as-is. It respects audit principles (generic parser, no dataset-specific logic) and is now fully covered by the new fixture tests.
- For Python:
  - AI-KG’s dedicated CLI tooling is justified because:
    - It serves regression, D3 export, and batch transformation use cases beyond the canvas UI.
    - It provides an independent cross-check of the same data used in canvas.
  - Possible simplifications (optional, not implemented):
    - Share a documented GraphData contract between TS and Python (schema already implies this, but could be made more explicit).
    - Gradually consolidate any purely structural transformations into a shared library or reference docs, keeping Python focused on regression and D3 export.
  - A0 and Universal Lean Startup:
    - Do not add Python CLI tooling unless you have external batch/analysis needs that mirror AI-KG’s D3/GraphRAG workflows.

**Net recommendation**

- Keep:
  - Single JSON-LD parser in canvas for all three file sets.
  - Unified Python CLI (`jsonld-universal`) for offline structural normalization.
- Avoid:
  - Introducing additional parser variants or dataset-specific parsers in the TypeScript canvas pipeline.
  - Duplicating dataset-specific Python parsers unless required by new non-canvas workflows.

**3. ADVICE: why one JSON‑LD parser in canvas, but a dedicated AI‑KG parser‑script in Python**

You quoted the split correctly:

> - Canvas / CI: one JSON‑LD parser for all three sets (no new parsers).  
> - Python: only AI‑KG uses a dedicated parser‑script; the other sets don’t need one for the use cases you described.

Here’s the reasoning and whether to simplify.

**Canvas / CI side (TypeScript)**

- **Single parser**: `parseJsonLd` in `canvas/src/lib/graph/jsonld.ts:17-118`.
  - It:
    - Collects `@graph` items into `nodes` and `edges`.
    - Uses `@context` to treat `@id`‑typed properties as edges (via `isIdPropertyKey`).
    - Strips `kg:` from IDs and handles both direct and reified edges.
  - It is intentionally **structural only**:
    - No special‑casing for “A0”, “Lean Startup”, or “AI‑KG”.
    - Consistent with `docs/knowgrph-codebase-audit-catalog.md:31-43` (no domain‑specific assumptions in core parsers).
- **Why this is good and simple:**
  - Any valid JSON‑LD that fits the graph shape can be loaded without adding new parsers.
  - All three of your file sets already conform, so they can share the same parser and validation pipeline.
  - Schema differences (styling, validation rules, ontology richness) live in the schema files, not in parser logic.

Given this, **I recommend keeping exactly this model**:

- One JSON‑LD parser in canvas.
- No extra TS parsers for AI‑KG, A0, or Lean Startup.

**Python side**

- Use the unified CLI (`python -m knowgrph_parser jsonld-universal`) for structural JSON/JSON-LD → `GraphData` normalization.
- If you need extra outputs (for example D3 `nodes`/`links` bundles) or dataset-specific batch transforms, keep that logic in separate offline modules and run it via `jsonld-universal --parser-module ...` so the core CLI remains stable.

- **Why A0 and Lean Startup do not (currently) need Python parsers:**

  - Their current role is:
    - Provide graph‑shaped inputs and schemas for visualization and RAG/decision‑tree thinking inside canvas.
    - Feed into structural validation and visualization, which are already covered by the TS pipeline.
  - There is no extra requirement yet to:
    - Export them into custom D3 formats.
    - Maintain historical regression artifacts independent of the canvas.
    - Run batch transformations that would be expensive in the browser.

**Should we simplify?**

- **On the TypeScript/canvas side:**
  - You are already at the simplest viable model:
    - A single JSON‑LD parser.
    - Generic validation and schema engines.
    - Dedicated but small fixture tests to catch regressions.
  - I would **not** introduce separate parser specs for AI‑KG / A0 / Lean, because that would:
    - Violate the “structural only” parser principle.
    - Add maintenance overhead with little benefit.

- **On the Python side:**
  - AI‑KG’s dedicated parser‑script is justified and valuable because it:
    - Encapsulates additional logic and regression checks.
    - Supports external consumers (D3, GraphRAG, etc.).
  - Potential simplifications (optional, future work, not needed now):
    - Clarify a shared “GraphData contract” between TS and Python (conceptually already there via schemas).
    - Keep Python focused on value‑added transformations (weights, layer assignments, pattern mining) and regression tests, not basic parsing.
  - I would **not** replicate AI‑KG’s Python pipeline for A0 or Lean Startup unless:
    - You need D3/POC HTML demos or heavy offline analysis for those as well.

**Net recommendation**

- Keep:
  - Single JSON‑LD parser in canvas for all three file sets.
  - AI‑KG Python scripts as the specialized, regression‑backed path for that domain.
- Do not:
  - Add separate TS parsers for each dataset.
  - Add unnecessary Python parsers for A0/Lean unless driven by new batch / external‑consumer requirements.
